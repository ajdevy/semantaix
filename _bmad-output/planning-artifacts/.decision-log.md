### Architecture decisions (Epic 11, 2026-05-22) — appended to architecture.md

- **Pipeline placement:** standalone `CalendarAvailabilityAnswerer` placed **before** `GroundedRagAnswerer` (deterministic answer, must short-circuit RAG; cheap opt-in gate = fast `handled=False`). Reuses existing scheduling-intent regex + `RussianNormalizer`. — chosen over a `scheduling_context` signal.
- **OAuth callback home:** **`api`** (co-located with token store/client; avoids cross-service handoff of the auth `code`); minimal HTML page; rate-limited; `state` validated against SQLite pending-state store. `web_ui` stays the admin UI.
- **Auth vs transport:** `google-auth-oauthlib` Flow (code exchange) + `google-auth` `Credentials.refresh()`, both **sync via `asyncio.to_thread`** (google-auth owns token transport); **`freeBusy` over injected httpx `AsyncClient` with explicit timeout**. Reject `google-api-python-client`.
- **New stores:** `.data/semantaix_calendar.db` (one-file-per-concern) → `calendar_project_settings`, `calendar_operator_tokens` (Fernet), `calendar_oauth_pending_state` (single-use TTL), `calendar_service_rules`.
- **Availability core:** pure clock-injected tz-aware `compute_availability(...)`; slot-fitting `[start, start+duration)`; sync repos via `to_thread`.
- **Service resolution (FR-22):** `calendar/service_resolver.py` using `RussianNormalizer.lemmas`.
- **Resilience (user-requested):** per-operator `asyncio.Lock` single-flight refresh; refresh-token expiry/revocation → reconnect + notify + incident + clear token; explicit httpx + google-auth timeouts → escalate; inbound rate-limit on callback + `/connect_calendar`; outbound 429 `Retry-After` + one bounded retry; one freeBusy call/question, no result caching (v1).
