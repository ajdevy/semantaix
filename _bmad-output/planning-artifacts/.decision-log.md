## Decisions (this session) — Calendar `/connect_calendar` auto-enables (Epic 11 follow-up), 2026-05-24

- **Drop the separate enable surface.** User directive: "we don't need a separate /calendar_on command, just make it on when using /connect_calendar." The operator `/calendar_on` command, the admin `/calendar_on @projectslug` command, and the `POST /calendar/projects/{id}/enable` endpoint are **removed**. `ApiClient.calendar_enable` is gone too.
- **Enable is implicit in `/connect_calendar`.** The `GET /calendar/oauth/callback` handler now calls `CalendarSettingsRepository.enable(...)` after the token upsert, atomically (best-effort): on success it flips `enabled=1` and records the connecting operator as the designated calendar operator; if `enable` raises, the callback returns a 500-class error and does NOT render the success page (the operator retries by re-running `/connect_calendar`). For an already-enabled project, the existing `project_timezone` / `lookahead_days` are preserved — only the designated operator may change. — FR-18, FR-21, story 11.08.
- **Disable + service config remain unchanged.** `/calendar_off` (operator + admin), `POST /disable`, `GET /settings`, and the service-rule routes (`POST .../services`, `DELETE .../services/{id}`) are intact. `/calendar_service add|remove` is unchanged. Re-enable after disable = the operator re-runs `/connect_calendar` (re-runs Google consent, replaces the token, re-flips `enabled=1`).
- **Disconnect is still operator-only.** `/disconnect_calendar` deletes the token; an admin attempting `POST /calendar/disconnect` with `actor_role=admin` still gets 403 (`admin_cannot_disconnect`).
- **Rationale.** Eliminates a redundant config step: the operator is already in front of Google consent, so requiring a second `/calendar_on` was friction without a permission benefit (an admin still cannot enable without an operator's consent — by design, since enable now requires real OAuth tokens). Disable stays explicit + shared (operator + admin) so an admin can pause an integration without the operator's involvement.

### Epic 12 — Unified Project Services Catalog (planning kickoff, 2026-05-24)

**Problem being resolved.** Today the bot has two disjoint "services" concepts and the same offering ("маникюр") must be described twice:
- **Catalog answer** (PR `4d85819`): "what services do you offer?" is answered from an LLM-built prose digest of operator-uploaded PDFs (`catalog_digests` / `catalog_digest.py` / `service_catalog_intent.py`). No structured services table.
- **Calendar service rules** (Epic 11): `calendar_service_rules` in `semantaix_calendar.db` holds operator-curated structured rows for `compute_availability` and the calendar `service_resolver`.

The two never reference each other. The user wants ONE canonical structured services list per project that powers BOTH the catalog answer AND the calendar.

**Decisions locked.**
- **Rename `calendar_service_rules` → `project_services`** in the same SQLite DB. Idempotent ALTER (table is empty in production — zero data risk). Columns: `id, project_id, name (REQUIRED), description, price_text, tags_json, duration_minutes, working_hours_json, service_days_json, date_exceptions_json, updated_at`. — FR-23, §6.
- **Calendar-eligibility = `duration_minutes IS NOT NULL`.** Calendar code filters on that; catalog-only rows are valid (name + maybe price/desc, no duration). Calendar-only rows and dual-use rows are also valid. — FR-23.
- **Canonical CRUD seam = `ProjectServiceRepository`** (sync `sqlite3` via `asyncio.to_thread`). `CalendarSettingsRepository`'s service-rule method names remain for one release as delegating aliases to ease the cutover. — FR-23.
- **Two editing paths converge on one upsert.** `/service add|remove|list` and a Russian NL dialog (`добавь услугу …`) both apply via `ProjectServiceRepository.upsert`. Same DB state regardless of input path. `/calendar_service` remains for one release as a deprecation-logged alias. — FR-24.
- **NL dialog mirrors the existing `nl_knowledge_ops` / `admin_nl_dialog` pattern.** New `ServicesNlOpsRepository` + `services_nl_op_sessions` table, state machine `pending_confirmation → confirmed | cancelled | expired`, TTL 600s, `confirm_token = secrets.token_urlsafe(16)`, atomic consume via `hmac.compare_digest`. New api endpoints behind `internal_service_token`. New bot module `services_nl_dialog.py`. — FR-24, §6.
- **Operator-AND-admin can edit services** on projects where they are registered as an operator (Epic 10 `operators` table). Non-registered senders are ignored silently with logged reason `unauthorized_services` — no session row, no token, no DM. — FR-24.
- **Admin gate narrowing (cross-reference Epic 11):** FR-24's admin must ALSO be a registered project operator — this is **narrower** than FR-21's plain admin gate for `/calendar_off`. Rationale: services are project-content the operator curates, not platform-level config; only operators-of-this-project should curate them. Admin disconnect remains as in FR-18 (operator-only).
- **Regex-based extraction in v1 (no LLM in the NL extraction path).** Ambiguous Russian input fails closed ("не понял, уточните"). LLM-based extraction is a future epic. — FR-24.
- **Catalog answer reads structured services first.** When `project_services` is non-empty for `ctx.project_id`, render as a labelled all-fields plain-text chunk and pass to the existing `answer_grounded` step (no extra LLM call vs today's digest). When empty, fall back to `_catalog_digest.get_digest(...)`. When both empty, existing `_skip(reason='catalog_empty')`. — FR-25.
- **Humanistic, question-tailored rendering.** Extend `grounding_system` with a Russian guidance rule so a "какие услуги?" question gets just the names and a "сколько стоит маникюр?" question gets just that service's price — no full dumps. — FR-25.
- **Audit logs keys, not values.** `services_nl_op_confirmed` records `trace_id, project_id, operator, op_type, payload_keys` — never the price/description content (they may carry sensitive data). — FR-24.
- **Delivery shape:** one combined PR for the Epic 12 code, but story-by-story inside (matching prior epics).

**Conflicts surfaced (and resolved in the 2026-05-24 validation pass — see "Validation resolutions" sub-section below).**

The original kickoff entry above claimed "(none)" — that claim was overconfident. The rubric and adversarial reviews caught three near-conflicts that the original framing walked past, all resolved during the validation pass:

1. **Permission-model asymmetry with Epic 11 (C1).** FR-21 splits permissions sharply: **disable** (reversible) is operator-AND-admin, but **disconnect** (destructive — deletes the operator-owned token) is operator-only. The original FR-24 kickoff declared "operator AND admin can add/edit/remove services" — collapsing that split and letting an admin perform destructive writes on operator-curated content. **Resolved:** `/service add` and `/service edit` are operator-AND-admin (non-destructive, analogous to enable/disable); `/service remove` is operator-only (destructive, analogous to disconnect) — admin attempting `/service remove` returns 403 `admin_cannot_remove_service`. Admin must additionally be a registered project operator (narrower than FR-21 — see admin-gate-narrowing bullet).
2. **Audit "keys not values" cargo-culted from FR-18's secret-handling rule (H5).** The original FR-24 said `services_nl_op_confirmed` logs only `payload_keys`, never values — borrowed wording from FR-18 / NFR-3's OAuth-token redaction rule. But `price_text` and `description` are non-secret operator-published content (the bot reads them back to every customer), and a keys-only audit makes the dispute path "who set маникюр's price to 9999 ₽ on 2026-05-30?" unanswerable. **Resolved:** log the full payload (same posture as today's `answer_traces`). NFR-3's OAuth-redaction scope is unchanged.
3. **"One release" deprecation window contradicted by the immediately-prior Epic 11 follow-up (H2).** FR-23 / FR-24 kept aliases "for one release" but the same 2026-05-24 decision-log entry (Epic 11 follow-up) **deleted** `/calendar_on` with no deprecation period — establishing a "one release = zero" precedent in this codebase. **Resolved:** aliases (`CalendarSettingsRepository.upsert_service_rule`, `/calendar_service`, `POST/DELETE /calendar/projects/{id}/services`) are pinned to the **Epic 13 cleanup PR (no later than 60 days after Epic 12 merge)**, and deprecated paths DM the operator a user-facing migration hint, not just log.

NFR-3's OAuth/secrets posture is preserved: `project_services` rows carry no per-operator credentials, so the encryption-at-rest mandate does not apply to this table. The only original framing that stands unchanged is "no prior decision asserts that `calendar_service_rules` must remain private to the calendar feature" — the rename / unification itself is consistent with the existing "config in DB, repository-owned" pattern.
