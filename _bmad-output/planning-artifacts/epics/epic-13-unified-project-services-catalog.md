# Epic 13: Unified Project Services Catalog

## Goal
Replace the two parallel "what services does this project offer?" sources (Epic 11's `calendar_service_rules` for the calendar; ad-hoc PDF prose for the catalog answer) with **one canonical structured `project_services` table per project** that powers BOTH the catalog answer ("какие услуги?") AND the calendar availability flow. The Epic 11 table is renamed to `project_services` and gains four catalog columns (`description`, `price_text`, `tags_json`, plus the existing scheduling fields); a row is **catalog-eligible always; calendar-eligible iff `duration_minutes IS NOT NULL`**. Two operator entry paths — `/service add|edit|remove|list` slash command and a Russian natural-language dialog (propose/confirm/cancel, mirroring `admin_nl_ops` / `admin_nl_dialog`) — converge on a single `ProjectServiceRepository.upsert` guarded by a per-`(project_id, lower(name))` `asyncio.Lock` (single-flight; last-writer-wins). The catalog answer **merges** structured rows with the existing `_catalog_digest.get_digest(...)` output via lemma-based dedup (`RussianNormalizer.lemmas`) so a partially-migrated project never silently loses content, and **renders natural Russian prose at the repository boundary with NO field labels** (`Название:`/`Цена:`/...) so the four-layer grounding gate cannot reject a legitimate answer for "label-leak hedging". `/service remove` is **operator-only** (preserving Epic 11's destructive-op rule); add/edit are shared with admin **only when that admin is also a registered project operator**. All `services_nl_op_*` events carry the full operator-typed payload (price/description are non-secret published content — the audit answer to "who set маникюр's price to 9999 ₽ on 2026-05-30?" must exist). Migration is idempotent (existence-check guards via `sqlite_master` + `PRAGMA table_info`) with a fresh-deploy path that creates `project_services` directly when no Epic-11 predecessor table exists.

## In Scope
- **Schema rename + new columns** in `.data/semantaix_calendar.db`: `calendar_service_rules` → `project_services` with added columns `description`, `price_text`, `tags_json`; new `UNIQUE(project_id, lower(name))` and `project_services_project_idx`. JSON column shapes pinned (`working_hours_json` per-weekday list of `[start,end]` windows; `service_days_json` lowercase 3-letter codes; `date_exceptions_json` ISO date strings).
- **Idempotent guarded migration + fresh-deploy path** — `sqlite_master` existence check before rename; `PRAGMA table_info` before each `ADD COLUMN`; `CREATE TABLE project_services` directly when neither old nor new table exists. Touches no other table in `semantaix_calendar.db`.
- **New `ProjectServiceRepository`** (sync `sqlite3`, dispatched via `asyncio.to_thread`) — canonical CRUD seam for catalog answer + calendar resolver; `upsert` keyed on `(project_id, lower(name))`; `list_calendar_eligible` filters on `duration_minutes IS NOT NULL`; raises `ProjectServiceNotFound`.
- **Per-`(project_id, lower(name))` `asyncio.Lock`** around `upsert` (single-flight, mirrors Epic 11's per-operator token-refresh lock); last-writer-wins.
- **Delegating aliases** on `CalendarSettingsRepository`'s service-rule methods (`upsert_service_rule`, `list_service_rules`, `delete_service_rule`) — call through to `ProjectServiceRepository` and emit `deprecation_warning_calendar_settings_service_rule` log; aliases removed in Epic 13 cleanup PR (≤60 days post-merge).
- **Canonical api endpoints** (behind `internal_service_token`): `POST /api/projects/{project_id}/services`, `GET /api/projects/{project_id}/services`, `DELETE /api/projects/{project_id}/services/{id}`, plus four NL ops endpoints (`POST .../services/nl-ops`, `POST .../services/nl-ops/{session_id}/confirm`, `POST .../services/nl-ops/{session_id}/cancel`, `GET .../services/nl-ops/latest-pending`).
- **Old `POST/DELETE /calendar/projects/{id}/services` endpoints** become delegating aliases that call the new handlers + log `deprecation_warning_calendar_services_endpoint`; endpoint deprecation is log-only (user-facing migration hint is DM'd at the command surface, story 13.03).
- **New `services_nl_op_sessions` table** in `.data/semantaix_nl_ops.db` (operator-scoped + project-scoped; mirrors `admin_nl_op_sessions`; TTL 600s; `confirm_token = secrets.token_urlsafe(16)`; atomic `consume()` via `hmac.compare_digest`; soft-delete with payload retained 30 days; full payload preserved through soft-delete for audit).
- **`services/api/app/services_nl_ops.py`** — `ServicesNlOpsRepository` + `parse_service_intent()` regex parser (ё/е + Cyrillic dash variants normalized via `RussianNormalizer`; at-minimum extracts all "must parse" examples in FR-24; "must fail closed" examples return `OP_UNKNOWN`).
- **Operator-only `/service remove` + shared `/service add|edit` + admin-must-be-registered-project-operator gate** (new `authorize_service_remove` helper; existing `authorize_calendar_config` reused for add/edit; admin attempting remove → 403 `admin_cannot_remove_service`).
- **`/service add|edit|remove|list` slash command** in `services/bot_gateway/app/calendar_commands.py` (start-of-message anchored; operator-gated via Epic 10 registry; `key=value` parser for `duration`/`days`/`hours`/`price`/`desc`/`tags`; catalog-only entries omit scheduling keys).
- **`/calendar_service` alias** kept as a deprecation-logged command; first invocation per operator DMs a one-time migration hint: "Команда `/calendar_service` устарела — используйте `/service` или просто напишите «добавь услугу …»".
- **`services/bot_gateway/app/services_nl_dialog.py`** — `handle_services_nl_message` dispatcher; start-of-message-anchored keyword triggers (regex `^\s*(добавь|добавьте|новая|создай|удали|измени)\s+услугу\b`); plain-text Russian preview (no MarkdownV2 / HTML; operator-supplied text escaped + length-capped at 200 chars); routes `да` / `/confirm <token>` and `нет` / `/cancel`; operator-gated via Epic 10 registry; non-registered → ignored with logged `unauthorized_services` and **no DM** (avoids customer-thread leakage).
- **At most one pending NL session per `(project_id, operator)`** — a new trigger atomically cancels the prior pending session (status flipped to `cancelled`) and DMs the operator: "ваш предыдущий запрос отменён, заменён новым".
- **Confirm endpoint verifies `session.originating_operator == sender`** — cross-operator replay returns 403 `not_session_owner`.
- **Full-payload audit** — every `services_nl_op_confirmed` / `services_nl_op_cancelled` / `services_nl_op_expired` event logs the complete payload (`trace_id, project_id, operator, op_type, name, description, price_text, tags, duration_minutes, working_hours_json, service_days_json, date_exceptions_json`). Operator-published service content is **non-secret** (the bot reads it to every customer); the FR-18 / NFR-3 secret-redaction rule remains scoped to OAuth tokens / encryption keys and is NOT applied to service content.
- **`services/api/app/services_render.py`** — new pure `render_project_service_prose()` rendering each row to natural Russian prose at the repository boundary with **NO field labels**. Working hours / service days / date exceptions resolved via `data/russian_calendar_terms.json` (new data file owned by story 13.01). Multi-window-per-day renders as `"пн 10:00–13:00, 14:00–19:00"`; date exceptions as `"закрыто: 1 января, 9 мая"`.
- **Catalog answerer cutover** — `GroundedRagAnswerer`'s catalog-query branch reads `project_services` first, calls the existing `_catalog_digest.get_digest(...)`, merges with lemma-based dedup (`RussianNormalizer.lemmas`; structured row wins on conflict; over-include is safer than under-include), and writes `answer_traces.source_id` = `project_services:<project_id>` (digest empty) / `catalog_digest:<project_id>` (structured empty) / `merged:<project_id>` (both contributed).
- **Extends `grounding_system` prompt** (via `project_prompt_repository`) with the FR-25 Russian humanistic + question-tailored rule ("Если клиент просто спрашивает, какие есть услуги — перечисли только названия, естественно и кратко. Если клиент спрашивает про цену, детали, описание или конкретную услугу — добавь только то, что он спросил. Не дампи всё подряд.").
- **Acceptance:** no field labels in customer-visible answer; single-service Q includes at most one `price_text` + one `description` (the resolved row); "какие услуги?" returns names only.

## Out of Scope
- LLM-based extraction of services from `/kb_add`-uploaded PDFs into `project_services` rows (future epic).
- Web admin UI for `project_services` CRUD (future epic; Epic 13 ships bot-only operator surface — slash + NL).
- Multi-operator selection / multi-calendar selection within a project (Epic 11 deferrals carry forward unchanged).
- Optimistic concurrency via `updated_at` precondition checks (last-writer-wins is acceptable for v1, mirroring Epic 11's calendar config writes).
- Booking / event creation / event modification (the calendar capability remains read-only per Epic 11; Epic 13 strictly broadens the **structured source of services**, not the calendar capability).
- Removal of any deprecated alias (`CalendarSettingsRepository.upsert_service_rule`, `/calendar_service`, `POST/DELETE /calendar/projects/{id}/services`) — that lands in the **Epic 13 cleanup PR (≤60 days post-merge)**.
- Adding new guardrail hedge phrases — story 13.06 audits `data/russian_hedges.txt` against typical price/duration phrasings ("от 2000 ₽", "от 60 минут") and only *removes* false-positive entries if needed (does NOT introduce new hedge categories).

## Exit Criteria
- **Migration is idempotent:** running it twice on the same DB is a no-op the second time (no `duplicate column name`, no `no such table`); existence checks via `sqlite_master` + `PRAGMA table_info` are exercised by an explicit "run twice" test.
- **Fresh-deploy path:** running the migration on a DB where neither `calendar_service_rules` nor `project_services` exists succeeds and produces `project_services` with the final schema, **without requiring any Epic 11 migration to have run first**. The other tables in `semantaix_calendar.db` (`calendar_project_settings`, `calendar_operator_tokens`, `calendar_oauth_pending_state`) are untouched (verified by snapshotting schemas before/after).
- **Slash-command happy path:** operator runs `/service add маникюр duration=60 days=mon-sat hours=10:00-19:00 price="от 2000" desc="…"` → row inserted; `GET /api/projects/{id}/services` returns it; `"какие услуги?"` answer is a clean humanistic Russian list of NAMES ONLY (no prices/descriptions, no field labels); `answer_traces.source_id` is `project_services:<id>` (digest empty) or `merged:<id>` (digest non-empty, dedup applied).
- **NL-dialog happy path:** same operator runs `"добавь услугу педикюр на 90 минут, вт–сб, 10:00–18:00, цена 2500"` → bot DMs a plain-text Russian preview with `/confirm <token>` and `/cancel`; reply `да` → row inserted; `"сколько стоит педикюр?"` includes the price but no other services' prices and no full data dump.
- **Permission rule:** admin attempts `/service remove маникюр` → 403 `admin_cannot_remove_service`; operator's `/service remove маникюр` succeeds. A non-registered sender attempting `/service add` or `добавь услугу …` is ignored with logged `unauthorized_services` and receives **no DM**.
- **NL session ownership:** operator B presents operator A's `confirm_token` → 403 `not_session_owner`; second pending session from the same `(project, operator)` cancels the prior atomically and DMs the operator the cancellation notice.
- **Calendar continuity (Epic 11):** customer asks `"можно записаться на маникюр в субботу в 15:00?"` → calendar resolver matches via FR-22 lemma against the calendar-eligible subset (`duration_minutes IS NOT NULL`); availability answer renders. Disabling calendar (`POST /calendar/projects/{id}/disable`) leaves the calendar question silently no-op while the catalog question still works (digest fallback or structured rendering as applicable).
- **Brownfield continuity (FR-25):** for a project that has only ever used `/kb_add` PDF uploads (empty `project_services`), the catalog answer continues to return the same content profile as before Epic 13 (digest-only path, `source_id` `catalog_digest:<project_id>`). After adding ONE row to a project with a 12-service digest, "какие услуги?" still returns up to 12 service names (merged + deduplicated), not just the one structured row; the new `source_id` is `merged:<project_id>`.
- **Deprecation:** deprecated `POST/DELETE /calendar/projects/{id}/services` endpoints + `/calendar_service` command + `CalendarSettingsRepository.upsert_service_rule` aliases succeed with the same data + emit `deprecation_warning_*` log; `/calendar_service` additionally DMs the operator a one-time migration hint (per-operator deduped).
- **No label leak:** the customer-visible answer for any catalog question contains none of the field-label substrings `Название:`, `Описание:`, `Цена:`, `Длительность:`, `Дни:`, `Часы:` (asserted by an explicit test in story 13.06).
- **`ruff check .` clean; `pytest --cov` shows 100% line coverage** on the new modules: `services/api/app/calendar/project_services_repository.py`, `services/api/app/services_nl_ops.py`, `services/api/app/services_render.py`, `services/bot_gateway/app/services_nl_dialog.py`, the new endpoint handlers in `services/api/app/main.py`, the `/service` command dispatcher in `services/bot_gateway/app/calendar_commands.py`, and the new `authorize_service_remove` helper in `services/api/app/calendar/authorization.py`; `pytest -m e2e` green.
- **Audit posture:** every successful `services_nl_op_confirmed` / `_cancelled` / `_expired` event logs the FULL payload (`name, description, price_text, tags, duration_minutes, working_hours_json, service_days_json, date_exceptions_json` + `trace_id, project_id, operator, op_type`); a release-readiness log-capture test confirms operator-published content is present (the FR-18 / NFR-3 secret-redaction rule does NOT apply to service content).
