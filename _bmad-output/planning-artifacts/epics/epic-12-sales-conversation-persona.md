# Epic 12: Sales Conversation Persona

## Goal
Add a `SalesPersonaAnswerer` that runs a consultative, multi-turn sales dialog end-to-end under a configurable sales persona (the persona name is operator-set per project via `hitl_runtime_config.get_bot_persona()` ‚Äî never hard-coded in the code or prompts). The answerer gathers customer intent (dates, headcount, vehicles, time, difficulty), autonomously dispatches the right video/photo/PDF with a one-line caption, lists configured services, looks up prices from the knowledge base, escalates to the operator when a price is unknown and **ingests the operator's reply back into the KB** so the bot learns it for next time, proposes a calendar slot via Epic 11, and nudges +1 day if the customer goes silent (skipping the nudge if the customer's target date has already passed). The capability is **always-on** for every project ‚Äî sales runs whenever the inbound text matches the sales-intent regex (or the chat has an existing funnel state). Services populate the `services` table through three input paths: slash commands, natural-language operator dialog, and automatic LLM extraction from `/kb_add` uploads. Discount handling is **out of scope** (deferred to a later epic).

The behavioral spec is two real Telegram dialogs extracted from screen-recordings: [2026-04-28_telegram_chat_danil.md](stories/epic-12/2026-04-28_telegram_chat_danil.md) (routes ‚Üí pricing per-vehicle ‚Üí –∫–∞–Ω—å–æ–Ω–∏–Ω–≥ concept ‚Üí date proposal ‚Üí next-day follow-up) and [2026-04-28_telegram_chat_darya.md](stories/epic-12/2026-04-28_telegram_chat_darya.md) (greeting with referral ‚Üí intent scoping ‚Üí media demo ‚Üí equipment Q&A ‚Üí tiered pricing menu).

## In Scope
- New SQLite store `.data/semantaix_sales.db` (idempotent schema, default-off bootstrap): `sales_conversation_state` (per-chat funnel state + collected intent JSON), `services` (the only offering table ‚Äî tours, baths, canyoning, paragliding all live here; optional description / tags), `client_materials` (video/photo/pdf/document attachments with tag-based selection; cached `telegram_file_id`; optional `source_operator_file_id` linking to a KB-uploaded file), `sales_followup_queue` (one nudge per chat with skip-if-stale semantics).
- New `services/api/app/sales/` package: `StateRepository`, `ServicesRepository`, `ClientMaterialsRepository`, `FollowupQueueRepository`, `price_lookup` (queries existing `rag_chunks` for price), `client_materials_selector` (tag-based pick), `client_materials_analyzer` (LLM client-sendability judgement on KB-uploaded files), `services_extractor` (LLM extraction of service-shaped entries from KB-uploaded files), `SalesPersonaAnswerer` (the funnel state machine), `proactive_followup` (scheduled job hosted in `services/scheduler`, promoting it from heartbeat to a real runner), `system_prompts/sales_*.txt` (per-stage persona prompts) + `system_prompts/operator_service_nl.txt` + `system_prompts/operator_services_extractor.txt`.
- New bot_gateway module: `operator_service_nl.py` ‚Äî LLM-driven NL classifier that maps operator free-text to the same `/service_*` handlers (story 12.02b).
- **Pricing ‚Äî knowledge-base-first, escalate-if-unknown:** on a price ask, `price_lookup` queries `rag_chunks` (lemma-overlap, same retriever as `GroundedRagAnswerer`); a hit is quoted verbatim with `source_id` in the trace; a miss escalates to HITL with `reason='price_unknown'` and a structured payload (`service`, `vehicle_type`, `hours`, original question) and replies to the customer "–£—Ç–æ—á–Ω—é —É –∫–æ–ª–ª–µ–≥ –∏ —Å—Ä–∞–∑—É —Å–æ–æ–±—â—É". The operator's HITL reply is delivered to the customer **and** picked up by the existing Epic-06 `knowledge_moderation` extractor ‚Üí moderation candidate ‚Üí operator approves ‚Üí RAG reindex. Next identical ask hits without escalating.
- **Pipeline wiring:** insert `SalesPersonaAnswerer` **before** `CalendarAvailabilityAnswerer` in `AnswerPipeline`. **Always-on activation:** the answerer takes the turn when a `sales_conversation_state` row exists for the chat OR when the inbound text matches a sales-intent regex (–ø—Ä–∏–≤–µ—Ç / —Ç—É—Ä / –∫–≤–∞–¥—Ä–æ / –±–∞–≥–≥–∏ / —ç–Ω–¥—É—Ä–æ / —Ü–µ–Ω–∞ / —Å—Ç–æ–∏–º–æ—Å—Ç—å / –¥–∞—Ç—ã / ...). No project-level enable, no `/sales_on` command, no `services_repo.count_active` check ‚Äî a project with zero services can still scope, look up prices via RAG, and propose calendar dates; the catalog turn handles the empty-catalog case by escalating to the operator.
- **Autonomous client materials dispatch:** new api endpoint `POST /sales/dispatch/material` (internal, service-token-gated) called by the answerer. Reuses `TelegramBotSender` plumbing from Epic 09 / operator `/send`. Files stay on local FS using the existing `operator_files` storage pattern. Bot sends video/photo/PDF with a one-line caption from `client_materials.caption`. `telegram_file_id` is cached on first send. Telegram-side delivery error ‚Üí textual fallback within the same turn; never silent.
- **KB-upload ‚Üí automatic client materials analysis:** every file added to the knowledge base (existing bot_gateway KB-upload flow) is automatically analyzed by `client_materials_analyzer` (no operator confirmation prompt). Suitable ‚Üí create a `client_materials` row with `source_operator_file_id` set; KB acknowledgement gains a one-line note "üìé –î–æ–±–∞–≤–ª–µ–Ω –≤ –º–∞—Ç–µ—Ä–∏–∞–ª—ã –¥–ª—è –∫–ª–∏–µ–Ω—Ç–æ–≤ (id=X)". Not suitable ‚Üí KB-only; no row created; no extra operator message. Override: the operator can always send the file as a Telegram message and use `/material` (reply mode) to register it directly, bypassing the analyzer.
- **Proactive follow-up:** one nudge at T+1 day after the last bot message goes unanswered. Skip-if-stale: if the customer's requested start datetime (`collected_intent.dates`) has already passed at fire time (project tz), the nudge is marked `status='skipped_stale'` and not sent. Quiet hours: project tz 21:00‚Äì10:00 ‚Üí defer to next morning. Customer reply between scheduling and firing ‚Üí `status='cancelled_replied'`.
- **Telegram operator commands** (bot_gateway, gated like Epic 09): `/service_add <name> [| description]`, `/service_list`, `/service_remove <id>`, `/material` (reply to a Telegram media message, optional caption arg), `/material_list`, `/material_remove <id>`, `/sales_state [@customer]`. All commands take the minimum required arguments; everything else optional. Validation errors return a one-line usage example. **Pricing has no command** ‚Äî prices live in the KB and grow via the price-unknown ‚Üí operator-reply ‚Üí moderation ‚Üí RAG loop.
- **Three input paths for services** populate the `services` table:
  1. **Slash commands** (story 12.02) ‚Äî `/service_add`, `/service_remove`, `/service_list`.
  2. **Natural-language operator dialog** (story 12.02b) ‚Äî operator says `"–¥–æ–±–∞–≤—å —É—Å–ª—É–≥—É X ‚Äî –æ–ø–∏—Å–∞–Ω–∏–µ"` / `"—É–¥–∞–ª–∏ —É—Å–ª—É–≥—É X"` / `"—Å–ø–∏—Å–æ–∫ —É—Å–ª—É–≥"` in DM; an LLM classifier maps to the same handlers as the slash commands. DRY: shared internal helpers, shared reply text.
  3. **Automatic LLM extraction from KB files** (story 12.05c) ‚Äî every `/kb_add` upload runs a `services_extractor` post-hook (in parallel with the materials analyzer from 12.05b); extracted service entries are added via `ServicesRepository.add` (idempotent on `(project_id, name)` ‚Äî never overwrites a manually-crafted description); the KB acknowledgement gains a `üì¶ –£—Å–ª—É–≥–∏ –¥–æ–±–∞–≤–ª–µ–Ω—ã: ...` line when any new entries are recorded. Confidential files (`/kb_add confidential`) never contribute.

## Out of Scope
- **Discount handling.** Deferred to a later epic. No discount commands, no discount fields, no discount turn. Discount asks fall through to HITL via the standard `low_confidence` / unknown-intent path.
- **Booking / payment.** Date proposal only; actual booking remains with the operator.
- **Voice messages.** No transcription, no TTS.
- **Image / video generation.** All client materials are operator-uploaded (directly via `/material` or auto-promoted from KB).
- **Sentiment / churn modeling.** Follow-up cadence is rule-based.
- **Web admin UI for sales settings.** Telegram commands only in v1; `/admin/*` web surface deferred.
- **Bulk catalog import** (CSV, YAML, file upload). One service at a time via `/service_add`.
- **Per-project enable command (`/sales_on` / `/sales_off`).** Sales is always-on for every project; there's nothing to enable or disable. Intent dispatch is the only gate.
- **Retroactive KB-to-materials scanning.** `client_materials_analyzer` fires only on the KB-upload code path, immediately after upload. No scheduled scan of existing files, no re-analysis command.
- **Re-relaxing the four-layer grounding stance.** Conversational turns (greetings, "—Å–ø–∞—Å–∏–±–æ") may bypass grounded-RAG verification, but any quoted **price** must come from a RAG hit (with a `source_id` in the trace) ‚Äî never from a free-form LLM call. Date proposals come from the Epic-11 calendar, not the LLM.

## Exit Criteria
- Clean `docker compose up`: `.data/semantaix_sales.db` exists with all 4 tables; no project has any `services` rows by default. `SalesPersonaAnswerer` is in the pipeline and **always-on** ‚Äî on a sales-intent message it greets and scopes even with an empty `services` catalog; on a non-sales-intent message it skips silently (no LLM call, no DB write) and the pipeline falls through to RAG/HITL unchanged.
- For a project with a few `/service_add` rows seeded: replaying the –î–∞–Ω–∏–ª inbound messages produces bot replies that (a) gather intent into `sales_conversation_state`, (b) on the price ask, **escalate with `reason='price_unknown'` if RAG is empty** OR **quote the RAG-found price (with `source_id` in the answer trace) if seeded**.
- After the operator answers a price-unknown ticket, a `knowledge_moderation` candidate is created automatically (via Epic 06 extractor ‚Äî no new code in this epic) such that approving it makes the next identical price ask answer without escalating.
- For the –î–∞—Ä—å—è dialog: bot sends an autonomous tour video (via `/material`-registered attachment) with the configured caption after the scoping turn, and the equipment list (from a `services` row description) after the equipment question.
- Customer silent ‚â• 24 hours ‚Üí exactly one nudge fires in project-tz daytime hours. Customer reply cancels (`status='cancelled_replied'`). Past intent date ‚Üí `status='skipped_stale'`, no message sent.
- Telegram send error on material dispatch ‚Üí textual fallback within the same turn, no retries, no second customer-visible message.
- Uploading a tour-catalog PDF to the KB ‚Üí `client_materials` row auto-created with `source_operator_file_id` set (from 12.05b) **and** any service-shaped entries are extracted into `services` (from 12.05c); KB acknowledgement includes both append lines: "üìé –î–æ–±–∞–≤–ª–µ–Ω –≤ –º–∞—Ç–µ—Ä–∏–∞–ª—ã –¥–ª—è –∫–ª–∏–µ–Ω—Ç–æ–≤ (id=X)" + "üì¶ –£—Å–ª—É–≥–∏ –¥–æ–±–∞–≤–ª–µ–Ω—ã: –ú–µ–¥–æ–≤–µ–µ–≤–∫–∞ –õ–∞–π—Ç, –ö–∞–Ω—å–æ–Ω–∏–Ω–≥".
